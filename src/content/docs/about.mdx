## What MarkdownPic is

MarkdownPic is being built as a **Markdown encyclopedia plus tool ecosystem**.

The site exists to answer the questions that matter once Markdown stops being a toy format and becomes part of a real publishing or documentation system:

- Which Markdown syntax is safest across renderers?
- Which editor is the right fit for a given workflow?
- How should Markdown assets, images, and exports be handled?
- How should Markdown support docs, blogs, notes, screenshots, and AI-assisted reuse?

The companion tool at [tool.markdownpic.com](https://tool.markdownpic.com/) solves a narrower execution problem: taking Markdown and converting it into polished long images and share assets.

## Why the site and tool are separate

They do different jobs.

- The **site** should rank, teach, compare, document, and answer long-lived search questions.
- The **tool** should convert intent into action once someone already knows they need a Markdown-to-image workflow.

Keeping them separate allows the site to become a stronger long-term reference asset without forcing every page to behave like a product landing page.

## What MarkdownPic covers

MarkdownPic is not trying to be a generic “developer blog.” It is organized around a smaller set of compounding themes:

### 1. Syntax that survives real use

Markdown looks simple until a file has to render correctly in GitHub, a docs site, an email preview, a PDF export, and a screenshot tool. Syntax pages focus on the forms that survive those boundaries with the least friction.

### 2. Editor and tooling decisions

Writers, developers, and documentation teams do not just need feature lists. They need to know which editor or workflow is the right fit for publishing, portability, privacy, image handling, collaboration, and AI use.

### 3. Markdown publishing systems

A file is not a system. MarkdownPic covers the move from isolated `.md` files into reusable content operations for docs, knowledge bases, blogs, static sites, newsletters, and share assets.

### 4. AI-era Markdown workflows

Markdown has become more valuable in the AI era because it is clean, portable, structured enough to reuse, and lightweight enough to manage. MarkdownPic treats Markdown as a canonical content layer for research, documentation, synthesis, and republishing.

## Editorial standard

MarkdownPic is being built around a strict rule: **pages should solve a real problem**.

That means a page should not exist unless it can do one or more of the following:

- clarify a syntax decision,
- reduce publishing friction,
- help someone choose an editor or workflow,
- explain a portability boundary,
- or provide an implementation path that saves time or avoids breakage.
