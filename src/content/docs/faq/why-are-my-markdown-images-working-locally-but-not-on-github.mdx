# Why are my Markdown images working locally but not on GitHub?

## What matters first

Because local preview and GitHub do not share the same file system assumptions.

Images often work locally but fail on GitHub when:

- the image file was never committed,
- the path is correct on your machine but wrong in the repository,
- filename case differs,
- the image sits outside the repo,
- or the local tool resolved a path GitHub cannot see.

The Markdown syntax can be valid while the repository path is still wrong.

## The first common problem: the file is not actually in the repo

A local editor can render:

- files on your machine,
- images outside the current folder,
- or paths that exist only in local preview context.

GitHub can only render what exists inside the repository path it knows about.

If the image was never committed, GitHub cannot load it.

## The second common problem: the relative path changed

A local preview may resolve the image from:

- the current file,
- the editor workspace,
- or another preview base path.

GitHub resolves the path from the committed Markdown file in the repository tree.

That is why the same path can look right locally and still break on GitHub.

## The third common problem: case sensitivity

A filename like:

- `Logo.png`

is not the same as:

- `logo.png`

Many local workflows hide this mistake. GitHub and other deployed environments often expose it immediately.

## The fourth common problem: the image is outside the repository

Paths that point to:

- parent directories,
- local downloads,
- desktop folders,
- or other machine-specific locations

are not portable to GitHub.

They may render locally because your machine can see them. GitHub cannot.

## The safest troubleshooting order

Check in this order:

1. confirm the image file is committed,
2. confirm the path from the Markdown file is correct in the repo,
3. check filename and folder case exactly,
4. confirm the image is inside the repository path,
5. then inspect whether the issue is GitHub rendering versus docs-site rendering.

That order catches most GitHub image failures fast.

## The healthiest long-term habit

Write image references for the repository and publishing system, not for the local machine.

That usually means:

- keeping images inside the repo,
- using deliberate relative paths,
- and testing with the same structure that GitHub will read.

Local success is helpful. Repository success is what matters.
