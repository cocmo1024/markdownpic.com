# Should you use root-relative paths in Markdown?

## What matters first

Use root-relative paths only when the Markdown belongs to a site with a stable, known root that owns the final routes.

Avoid them when the file needs to stay portable across:

- repositories,
- local previews,
- note apps,
- docs migrations,
- or multiple publishing systems.

Root-relative paths are convenient inside one site. They are often brittle outside it.

## What root-relative means

A root-relative path starts from the site root, for example:

```md
[Docs](/guides/setup/)
```

That is different from a file-relative path such as:

```md
[Docs](./setup.md)
```

One assumes a site root. The other assumes a file location.

## When root-relative paths are healthy

Root-relative paths are usually healthy when:

- the site root is stable,
- the content is already part of one publishing system,
- routes matter more than repository structure,
- and local portability is not the primary goal.

They can simplify navigation in large sites where page routes are the canonical truth.

## When root-relative paths become debt

They become debt when:

- the Markdown also lives in a repository as reusable source,
- the file is copied into other systems,
- local preview tools do not share the same root assumption,
- or the site later changes base path or deployment structure.

At that point the path is no longer simple. It is coupled to one publishing environment.

## The practical rule

Use root-relative paths when the site route is the real source of truth.

Use file-relative paths when the repository or docs tree is the source of truth.

If the team cannot say which of those is primary, the content system still needs a linking policy.

## What breaks during migration

Root-relative paths often fail during migrations because they hide a site assumption inside ordinary-looking Markdown. A move from `/docs/` to `/help/`, a deployment under a subpath, or a copy into a package README can turn a clean link into a broken one.

Before using root-relative paths heavily, confirm:

- whether the site will always deploy from the domain root;
- whether local preview tools resolve `/` the same way production does;
- whether the same files are reused in README, package, or CMS contexts;
- whether redirects or base-path settings will be controlled by the docs team.

If those answers are uncertain, file-relative links are usually safer until the publishing contract is stable.
