# Should you use relative or absolute links in Markdown?

## What matters first

Use **relative links** when the Markdown lives inside a stable repository, docs tree, or content system that moves together.

Use **absolute links** when the content will travel across sites, systems, or publishing surfaces that do not share one consistent file structure.

The right question is not "Which style is cleaner?" It is:

**"Which environment owns the truth of this content?"**

## When relative links are stronger

Relative links are usually stronger when:

- the files live in one repository,
- the docs structure is stable,
- branches and forks should keep links working,
- and portability inside that content tree matters most.

That is why relative links are often right for READMEs and structured docs repos.

## When absolute links are stronger

Absolute links are usually safer when:

- the content is copied into multiple systems,
- the destination lives outside the current repository,
- publishing rewrites local paths aggressively,
- or the business needs one canonical public URL regardless of source location.

They are heavier to maintain, but often more honest about where the content actually lives.

## The common mistake

The common mistake is choosing relative links because they look elegant, then expecting them to survive:

- site generation,
- CMS import,
- docs migrations,
- image export,
- or AI-assisted reuse workflows.

At that point the link problem is not syntax style. It is content portability.

## A simple decision rule

Choose relative links when the source tree is the system of record.

Choose absolute links when the published site is the system of record.

If the content has to serve both, the team needs a deliberate linking policy instead of ad hoc habits.
