## Quick answer

**CommonMark** is the closest thing Markdown has to a formal baseline specification. **GitHub Flavored Markdown (GFM)** starts from that baseline and adds practical features such as tables and task lists. A **renderer** is the actual implementation that decides what the Markdown file turns into.

That means two files can both be called “Markdown” while still behaving differently, because the renderer is not identical.

## Why this difference matters

People usually discover the distinction only after something breaks:

- a table works on GitHub but not in another renderer,
- a task list becomes plain text outside GFM,
- line breaks behave differently after copy-paste,
- or a tool accepts raw HTML and another strips it.

If the content is meant to travel, you need to know which layer you are relying on.

## A practical model

Use this mental model:

- **CommonMark** for the baseline grammar and predictable core behavior.
- **GFM** for GitHub-style extensions that many developers now expect.
- **Renderer behavior** for everything implementation-specific, including sanitization, HTML handling, image resolution, and extension support.

## What is usually safest

When portability matters most:

- favor CommonMark-safe syntax where possible,
- treat GFM features as opt-in conveniences,
- and assume renderer-specific features are local contracts, not universal Markdown.
