## Quick answer

Standard Markdown cannot force a link to open in a new tab.

If you need `target="_blank"`, you have crossed from Markdown into HTML or renderer-specific behavior. That means the direct workaround is usually an HTML anchor tag, but the better architectural question is whether new-tab behavior belongs in the source at all.

## Why this keeps confusing people

People expect Markdown links to support common browser behavior flags because the rendered output is often HTML.

But Markdown link syntax only describes the link itself:

```md
[Example](https://example.com)
```

It does not define HTML attributes such as:

- `target`
- `rel`
- `class`
- `data-*`

Those belong to the HTML layer or to a renderer that adds its own syntax extensions.

## The honest fallback

If you truly need new-tab behavior in the source, the fallback is HTML:

```html
<a href="https://example.com" target="_blank" rel="noopener noreferrer">Example</a>
```

That works because HTML supports the attributes Markdown does not.

The tradeoff is that the source is now less portable, and some renderers may sanitize or reject the attributes.

## Why source-level `target="_blank"` is often the wrong answer

Many teams add new-tab behavior in Markdown because they are trying to solve a site-level UX decision with content syntax.

That is usually backward.

In a well-structured publishing system, external-link behavior is often better handled by:

- the renderer,
- a site component layer,
- or a global external-link policy.

That keeps Markdown source cleaner and avoids mixing presentation behavior into every file.

## When HTML is justified

Using HTML in source is justified when:

- the publishing surface is controlled,
- the external link behavior is intentional and important,
- the renderer preserves the attributes,
- and the team accepts the portability cost.

This is common in site-authored docs that are not meant to round-trip through many Markdown environments.

## When HTML is not worth it

HTML is usually not worth it when:

- the same file should stay portable across GitHub, docs, notes, and export,
- the team wants source to remain clean and readable,
- or the desired behavior is really a site-wide external-link policy rather than a page-level exception.

In those cases, forcing the behavior into source usually creates more maintenance debt than value.

## The accessibility and UX issue

Opening links in a new tab is not automatically better UX.

It can:

- disrupt expected navigation behavior,
- confuse keyboard and screen-reader users,
- and create inconsistent behavior when some links open new tabs and others do not.

That is one more reason to treat this as a policy decision, not as a syntax trick.

## The best team rule

A healthy rule is:

- use plain Markdown links by default,
- handle external-link behavior in the site or renderer when possible,
- use HTML only where the source must truly encode the behavior,
- and if HTML is used, include `rel="noopener noreferrer"` alongside `target="_blank"`.

That rule preserves portability where possible and keeps exceptions explicit where needed.
