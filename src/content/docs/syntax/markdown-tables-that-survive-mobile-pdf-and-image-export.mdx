Markdown tables usually fail because authors ask one format to serve three incompatible jobs:

- a desktop doc page,
- a phone screen,
- and a fixed visual export.

## What matters first

Keep tables only when the rows are short and the comparison genuinely depends on columns. If the content is verbose, turn the table into sections, cards, or repeated comparison blocks instead.

## Standard example

```md
| Editor  | Best fit              | Weak point              |
| ------- | --------------------- | ----------------------- |
| Typora  | Focused writing       | Less repo-native        |
| VS Code | Publishing workflows  | More setup              |
| Obsidian| Knowledge systems     | Publishing is less direct |
```

That works because the cells are short and the reader can compare three attributes quickly.

## The practical limit

A Markdown table stops being robust when:

- cells contain paragraphs,
- code examples become long,
- URLs are unbreakable,
- or the number of columns exceeds what a small viewport can display without horizontal scrolling.

Once that happens, the table is doing too much.

## The mobile rule

If the reader has to pan sideways to understand a decision, the table should probably be rewritten.

Good Markdown tables:

- shortlist a few tools,
- compare a few attributes,
- or summarize narrow differences.

Bad Markdown tables:

- try to hold full decision logic,
- long prose,
- or implementation instructions.

## PDF and image export implications

PDF and image export are less forgiving than web pages because the layout is more fixed. A table that is barely readable in HTML usually becomes actively bad in screenshots or long images.

Use a short table for scanning, then explain the decision below the table in normal Markdown prose. That gives you both fast comparison value and export resilience.

## Recommended operating rule

- Keep tables narrow.
- Keep cells short.
- Avoid putting core reasoning only inside the table.
- Rewrite long logic as prose underneath the table.

That is the difference between a table that helps scanning and a table that collapses in export.
