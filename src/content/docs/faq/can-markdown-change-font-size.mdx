# Can Markdown change font size?

## What matters first

No. **Standard Markdown does not have a native font-size syntax.**

If text size must change, the solution usually lives in one of these layers:

- HTML inside Markdown,
- CSS or theme styling,
- a renderer-specific extension,
- or a downstream export tool.

That means "change font size in Markdown" is really a Markdown-versus-presentation question.

## Why this confuses so many people

Markdown looks simple enough that people expect it to include common text-formatting controls such as:

- font size,
- alignment,
- color,
- and columns.

But Markdown was designed for readable structure, not detailed typography. It handles headings, emphasis, links, images, code, and lists well. It does not aim to be a layout language.

## What standard Markdown can do instead

If the goal is visual hierarchy, Markdown already gives you:

- headings,
- paragraphs,
- lists,
- blockquotes,
- and code formatting.

Often the healthiest fix is not custom font size at all. It is choosing the right structural element and letting the renderer style it.

## When HTML is the fallback

If the target renderer allows inline HTML, authors sometimes use a tag like this:

```html
<span style="font-size: 1.125rem;">Important note</span>
```

That may work in one environment, but it creates tradeoffs:

- portability drops,
- sanitizing renderers may strip the style,
- and the source file becomes more presentation-specific.

This is why HTML should be a narrow escape hatch, not the default authoring model.

## The better long-term rule

If the file needs to travel across:

- GitHub,
- docs sites,
- CMS fields,
- chat tools,
- email,
- or visual export pipelines,

then font-size control should usually live **outside** the Markdown source.

That keeps the file readable, reusable, and easier to transform later.

## Where font size should live instead

In a healthier system:

- Markdown carries the content and structure,
- CSS or templates handle typography,
- and image or PDF workflows handle visual packaging after the source is written.

This is the same reason Markdown remains useful at scale: it works best as canonical content, not as hand-built page layout.

## Use this rule

If the text needs semantic hierarchy, use Markdown structure.

If the text needs visual styling, use the renderer or export layer.

Only use inline HTML when:

- the target renderer is known,
- the style is necessary,
- and portability is not the top priority.
