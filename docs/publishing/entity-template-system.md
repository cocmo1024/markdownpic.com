# MarkdownPic Template System

This repository is designed for long-term, AI-assisted publishing, but the live site must read like a serious Markdown reference system, not a thin keyword site.

Every substantive page should solve a real Markdown, editor, publishing, or workflow problem.

## Publishing bar

Every page that targets search traffic must do all of the following:

1. target a real Markdown problem or decision,
2. state what decision the page helps the reader make,
3. explain fit, poor fit, tradeoffs, and implementation consequences,
4. move the reader forward with the right related page,
5. avoid overlapping too heavily with an existing page in the same cluster.

General depth targets:

- `seed`: usually 1,000+ words
- `growing`: usually 1,300+ words
- `cornerstone`: usually 1,700+ words

Do not publish:

- glossary-only pages with no practical decision value,
- syntax pages that repeat basic examples without boundary cases,
- editor pages that collapse into feature lists,
- workflow pages that describe ambition but not operating models.

## Shared frontmatter

Use these fields on every substantive page:

- `title`
- `description`
- `referenceType`
- `adProfile`
- `commercialIntent`
- `reviewCadence`
- `primaryKeyword`
- `searchIntent`
- `decisionStage`
- `targetRoles`
- `contentStatus`
- `problemSolved`
- `readerTakeaway`
- `keyQuestions`
- `notFor`
- `updateTriggers`

## Universal long-form structure

Use this shape unless the page has a strong reason to vary:

1. `Quick answer`
2. `Why this problem exists`
3. `Where the common advice breaks`
4. `Decision logic`
5. `Fit, poor fit, and tradeoffs`
6. `Implementation or publishing implications`
7. `Checklist or next-step rule`
8. `Related pages to read next`

## Syntax template

Use for link, image, code, table, embed, and portability pages.

Required sections:

1. Quick answer
2. Safe syntax example
3. What varies across renderers
4. Common failure modes
5. Decision rule
6. Related pages

## Editor template

Use for editor profiles or editor-category pages.

Required sections:

1. Quick fit statement
2. Who the tool is good for
3. Workflow strengths
4. Workflow weaknesses
5. Image and publishing implications
6. Related comparisons

## Publishing template

Use for static-site, docs, blog, newsletter, and distribution-system pages.

Required sections:

1. Quick answer
2. Canonical content model
3. Distribution layers
4. Failure points
5. Recommended operating model
6. Related workflow and syntax pages

## Workflow template

Use for AI, reuse, screenshot, and documentation operations pages.

Required sections:

1. Quick answer
2. Canonical source layer
3. Working layer
4. Distribution layer
5. Operational mistakes
6. Checklist
7. Related pages

## Comparison template

Use for shortlist-stage pages with high commercial intent.

Required sections:

1. Quick verdict
2. What should decide the shortlist
3. Fit by user type or workflow
4. Weaknesses and hidden costs
5. Who should choose what
6. Related next-step pages
