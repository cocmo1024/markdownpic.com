## Quick answer

Standard Markdown tables do **not** support `rowspan` or `colspan`.

If a table truly needs merged cells, the correct answer is usually to switch from pipe-table syntax to an HTML table. That works because merged cells are an HTML table feature, not a Markdown table feature.

The practical cost is portability. Once you move into HTML tables, you are depending on renderer support, sanitization rules, CSS defaults, export behavior, and authoring discipline that plain Markdown tables usually avoid.

## The boundary checked April 15, 2026

The standards boundary is straightforward:

| Source | Current signal | What it means |
| --- | --- | --- |
| [CommonMark 0.31.2](https://spec.commonmark.org/0.31.2/) | Core CommonMark does not define pipe-table syntax as part of the base spec | Merged cells are far outside core Markdown expectations |
| [GitHub Flavored Markdown spec](https://github.github.com/gfm/) | GFM tables support header and cell layout in pipe syntax, but not merged rows or columns | A normal Markdown table cannot express rowspan or colspan |
| [GitHub Docs](https://docs.github.com/en/get-started/writing-on-github) | GitHub teaches basic Markdown table syntax, not merged-cell table semantics | If you need merged cells, you have already crossed into HTML territory |

This is why people keep searching for "markdown colspan" or "markdown rowspan" and end up confused. The syntax they want belongs to HTML, not Markdown.

## Why Markdown tables stop early

Pipe tables are intentionally simple. They are good at:

- simple header rows,
- regular rectangular data,
- quick comparison tables,
- and source that remains readable in plain text.

They are poor at:

- layout-heavy comparison charts,
- grouped headers with merged cells,
- multi-row category labels,
- and tables that are trying to behave like a spreadsheet or magazine layout.

That limitation is not a bug. It is one of the reasons pipe tables remain readable as source.

## The correct fallback

If merged cells are truly required, use HTML directly:

```html
<table>
  <tr>
    <th rowspan="2">Category</th>
    <th colspan="2">Output</th>
  </tr>
  <tr>
    <th>Portable</th>
    <th>Renderer-specific</th>
  </tr>
  <tr>
    <td>Tables</td>
    <td>Pipe table</td>
    <td>HTML table with merged cells</td>
  </tr>
</table>
```

That is more honest than pretending Markdown can express the same structure.

## When HTML tables are justified

HTML tables are justified when:

- the merged structure carries real meaning,
- grouped headers materially improve comprehension,
- the publishing surface is controlled and known,
- and the organization accepts a less portable source format in exchange for better output.

This is common in technical comparison matrices, standard mappings, and docs pages where merged headers carry the logic of the table.

## When HTML tables are not worth it

Switching to HTML is usually a bad trade when:

- the merge is mostly cosmetic,
- the same information can be expressed with repeated labels,
- the content must survive GitHub, docs exports, CMS reuse, and image capture,
- or the authoring team needs source files to stay simple for non-HTML-heavy contributors.

In those cases, repeated labels often beat clever layout.

## The better portability move

If the real goal is portability, use one of these instead:

- repeat the category label instead of merging it,
- split one complex table into two smaller tables,
- convert layout intent into headings plus smaller tables,
- or move commentary out of the table and back into prose.

The real question is not "How do I hack colspan into Markdown?" It is whether you want portable source or richer layout.

## Implementation rule for teams

A good team rule is:

- use pipe tables for canonical Markdown,
- allow HTML tables only when merged cells materially improve comprehension,
- document which renderers are expected to preserve the HTML,
- and test export paths before relying on the result.

That rule stops table markup from slowly turning the whole content system into hidden HTML.
