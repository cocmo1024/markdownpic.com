# Why are my Markdown relative links not working?

## Quick answer

Relative links usually fail because the destination is being resolved from a different base path than you expected.

That often happens when:

- the current file moved,
- the published route no longer matches the repo path,
- the renderer rewrites `.md` files into HTML routes,
- path casing differs across environments,
- or the content was copied into a system that no longer understands the original folder structure.

The syntax can be correct while the environment is wrong.

## Relative links only make sense relative to something

A normal relative link looks like this:

```md
[Guide](./guide.md)
```

That link is valid only if the renderer resolves it from the current file in the way you expect.

The fragile assumption is believing that all Markdown tools share the same base-path logic.

## The most common failure: repo paths are not site routes

A relative link may work inside:

- a GitHub repository,
- a local docs folder,
- or a note app

and still fail after publishing because the final site route is now:

- HTML instead of `.md`,
- nested under a docs base path,
- or generated from frontmatter and framework rules.

That is why relative links are often a publishing problem, not a syntax problem.

## Another common failure: file extensions and route rewriting

Some systems want:

- `./guide.md`

while others want:

- `/guide/`
- `/guide`
- or a framework-specific route.

If your source moves across systems, one relative-link form may stop being stable enough.

## Case sensitivity still breaks real sites

Many local workflows are forgiving. Many deployed environments are not.

So links can break because:

- `Guide.md` is not `guide.md`,
- folder names changed,
- or imported assets use different case than the published file tree.

These failures look small and waste a lot of time.

## The safest troubleshooting order

Check in this order:

1. confirm the destination file really exists,
2. confirm the path is correct from the current file,
3. inspect the final published route,
4. test whether the system expects `.md`, `.html`, or route-style links,
5. then check casing and path rewrites.

That order usually finds the actual break faster than changing syntax forms blindly.

## When relative links are still the right answer

Relative links remain strong when:

- the content lives in a stable repo tree,
- the publishing system respects source structure,
- and the team values portability inside that repository.

They become weak when the content is constantly copied, transformed, or published into systems with different routing assumptions.
