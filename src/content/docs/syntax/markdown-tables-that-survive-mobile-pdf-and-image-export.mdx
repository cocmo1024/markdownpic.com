Markdown tables usually fail because authors ask one format to serve three incompatible jobs:

- a desktop doc page,
- a phone screen,
- and a visual export.

## Quick answer

Keep tables only when the rows are short and the comparison really depends on columns. If the content is verbose, turn the table into sections, cards, or repeated comparison blocks instead.

## The practical limit

A Markdown table stops being robust when:

- cells contain paragraphs,
- code examples become long,
- URLs are unbreakable,
- or the number of columns exceeds what a small viewport can display without horizontal scrolling.

## The mobile rule

If the reader has to pan sideways to understand a decision, the table should probably be rewritten.

Good Markdown tables:

- shortlist a few tools,
- compare a few attributes,
- or summarize narrow differences.

Bad Markdown tables:

- try to hold full decision logic,
- long prose,
- or implementation instructions.

## Export implication

PDF and image export are less forgiving than web pages because the layout is more fixed. A table that is barely readable in HTML usually becomes actively bad in screenshots.

Use a short table for scanning, then explain the decision below the table in normal Markdown prose. That gives you both fast comparison value and export resilience.
