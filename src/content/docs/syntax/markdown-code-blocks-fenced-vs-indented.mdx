## Quick answer

Use **fenced code blocks** for almost all modern Markdown work. Indented code blocks still exist, but fenced blocks are clearer, easier to edit, and better aligned with syntax-highlighting workflows.

## Standard examples

### Fenced code block

````md
```js
console.log('Hello, world');
```
````

### Indented code block

```md
    console.log('Hello, world');
```

## Why fenced blocks are the better default

Fenced blocks win because they:

- are explicit in source,
- support language hints,
- are easier to move around during editing,
- and create fewer ambiguities when mixed with lists, quotes, and surrounding prose.

If the file will live in docs, a site repo, or a publishing system, fenced blocks are usually the safest operational choice.

## Where indented blocks still appear

Indented blocks still work in many Markdown contexts, especially in older writing habits or very plain documents. But they create more edge cases when mixed with:

- nested lists,
- quoted content,
- copied snippets,
- and editors that auto-format indentation.

That is why they are no longer the strongest default for serious Markdown publishing.

## Language labels and syntax highlighting

If you want reliable syntax highlighting, fenced blocks are the right tool because they allow a language label:

````md
```python
def add(a, b):
    return a + b
```
````

Indented blocks usually do not carry that same clean, explicit language signal.

## Common failure modes

The typical mistakes are:

- forgetting to close a fenced block,
- using the wrong number of backticks when the content contains backticks,
- or relying on indented blocks inside heavily nested content where indentation already has meaning.

The fix is simple: default to fenced blocks and keep them visually obvious in the source.

## Recommended operating rule

- Use fenced blocks for almost everything.
- Add a language label when the renderer benefits from it.
- Use indented blocks only if you have a strong reason and know the surrounding structure is simple.
