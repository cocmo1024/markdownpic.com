# Why does Markdown render differently on GitHub and VS Code?

## Quick answer

Because GitHub and VS Code are not the same renderer.

They may differ in:

- supported extensions,
- image and link resolution,
- heading anchors,
- task-list behavior,
- raw HTML handling,
- and preview configuration.

The file can be valid Markdown while the rendered output still differs.

## The first mistake: treating "Markdown" as one uniform thing

Many authors think:

**"If it works in one Markdown viewer, it should work everywhere."**

That is not how Markdown works in practice.

GitHub, VS Code preview, docs frameworks, and note apps each apply their own rendering rules on top of Markdown parsing.

## Where GitHub and VS Code diverge most often

The most common differences show up in:

- task lists and GFM-specific features,
- anchor generation,
- emoji shortcodes,
- raw HTML support,
- local image paths,
- and link behavior tied to the repository versus the local workspace.

That is why the same file can look polished in one environment and awkward in another.

## Paths are a common hidden cause

VS Code may preview files relative to:

- your local workspace,
- local filesystem access,
- or extension-specific preview behavior.

GitHub resolves paths from the committed repository tree.

That difference alone explains many image and link mismatches.

## Extensions and preview settings matter too

VS Code can be influenced by:

- installed extensions,
- preview configuration,
- theme and CSS overrides,
- or workspace-specific tooling.

GitHub will ignore most of that and render through its own rules.

So sometimes the issue is not Markdown at all. It is your editor environment.

## The safest troubleshooting order

Check in this order:

1. does the raw syntax follow a portable form,
2. does the file depend on GitHub-flavored features,
3. are paths valid in the committed repository,
4. is VS Code preview being affected by extensions or local assumptions,
5. does GitHub sanitize or ignore HTML that the editor preview tolerated?

That usually shows whether the difference is a syntax problem or a renderer problem.

## The healthiest long-term rule

Write for the most important rendering environment, then test the others deliberately.

If the file must survive both GitHub and other platforms, prefer:

- standard syntax,
- known GFM features only when intentional,
- stable paths,
- and minimal dependence on custom preview behavior.

Markdown is portable only when the team treats renderer differences as real.
