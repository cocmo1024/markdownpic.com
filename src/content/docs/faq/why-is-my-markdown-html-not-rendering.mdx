# Why is my Markdown HTML not rendering?

## What matters first

Raw HTML in Markdown can fail for two very different reasons:

1. the Markdown parser treats the HTML differently than you expected,
2. the renderer or platform sanitizes, strips, or ignores the HTML afterward.

That is why valid HTML can still fail in a Markdown workflow.

## Markdown allowing HTML is not the same as the platform allowing HTML

Many Markdown systems will parse raw HTML.

But the platform may still:

- remove unsafe tags,
- strip attributes,
- ignore scripts and styles,
- or block layout-heavy HTML patterns entirely.

So the right question is not only "is this valid HTML?" but also "does this renderer allow it?"

## HTML blocks can change Markdown parsing

Raw HTML can also affect how surrounding Markdown is parsed.

That means you can run into problems with:

- blank lines,
- indentation,
- block boundaries,
- or Markdown that you expected to render inside an HTML block.

Sometimes the issue is not sanitization. It is the parser switching modes.

## The most common failure patterns

HTML in Markdown often fails because of:

- unsupported tags like script-heavy or embed-heavy elements,
- inline styles getting stripped,
- raw HTML blocks swallowing nearby Markdown unexpectedly,
- custom classes that only work on one docs stack,
- or an editor preview that tolerates HTML a publishing surface later sanitizes.

These are renderer problems more often than syntax problems.

## The safest troubleshooting order

Check in this order:

1. simplify the HTML to the smallest possible example,
2. test whether the target renderer allows that tag at all,
3. remove styles, scripts, and custom attributes,
4. check whether block-level HTML changed Markdown parsing around it,
5. then compare local preview with the actual destination platform.

That usually shows whether the failure is parsing, sanitization, or platform policy.

## The healthiest long-term rule

Use raw HTML in Markdown only when:

- the destination platform is known,
- the HTML is simple,
- and portability is not your top requirement.

The more layout or behavior you force into HTML, the less portable the Markdown source becomes.
