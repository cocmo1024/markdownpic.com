# Can Markdown link to a local file?

## Quick answer

Sometimes, yes, but only in environments that can actually access that local path.

A Markdown link can point to a local file, but that does **not** mean the link is portable, shareable, or useful outside your own machine or controlled environment.

That is why local file links often work in:

- note apps,
- local previews,
- or private internal systems

and then fail everywhere else.

## Markdown can point to a path, but the renderer still decides

A link such as:

```md
[Open notes](./notes/today.md)
```

or a file URL may be readable to one tool and meaningless to another.

Markdown can express the link. The environment still decides whether the destination is reachable.

## Why local file links are usually fragile

Local file links break when:

- the file lives only on your machine,
- the folder structure differs for another user,
- the content is published to GitHub or a docs site,
- the target platform blocks local file access,
- or the path only made sense inside one app's workspace model.

This is why local links are often fine for personal notes and poor for public documentation.

## When local file links are acceptable

Local file links are most acceptable when:

- the workflow is private,
- the environment is controlled,
- all users share the same storage model,
- and portability is not the goal.

That can be true in personal knowledge bases or tightly managed internal systems.

## When to avoid them

Avoid local file links when the Markdown is meant for:

- GitHub,
- public docs,
- exported HTML,
- shared README files,
- or any workflow where another machine must open the same link reliably.

In those cases, local links are usually the wrong abstraction.

## The practical rule

Use local file links only when the file system is part of the intended environment.

If the content is going to travel, use repository-relative, site-relative, or public URLs instead.

That keeps the Markdown useful outside your own machine.
