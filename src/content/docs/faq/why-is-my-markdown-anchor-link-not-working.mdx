# Why is my Markdown anchor link not working?

## What matters first

A Markdown anchor link usually breaks because:

1. the heading text changed,
2. the renderer generated a different slug than expected,
3. custom IDs were stripped or ignored,
4. the file published to a different route,
5. or the link targets a fragment rule Markdown itself does not standardize.

The page may still load correctly while the jump fails. That usually means the fragment target, not the path, is the real problem.

## Markdown does not define one universal anchor rule

Markdown syntax can create a link with a fragment:

```md
[Jump to section](#installation)
```

But Markdown does **not** define one universal way every renderer must generate heading IDs.

That is why the same heading can produce different anchors across:

- GitHub,
- static site generators,
- note apps,
- docs frameworks,
- and HTML export tools.

## The most common reason: the heading changed

If the heading text changes, the generated anchor often changes too.

That means links break when:

- wording is edited,
- punctuation is added or removed,
- duplicated headings force suffixes,
- or the title casing changes in a renderer that normalizes slugs differently.

Anchor links are cheap to create and easy to break later.

## The second common reason: slug rules are different

Different renderers handle:

- punctuation,
- emoji,
- non-Latin characters,
- duplicate headings,
- and special symbols

differently.

So a fragment that worked in GitHub may fail in a docs site because the slugging logic is not identical.

## The third common reason: the publishing system rewrote the page

Even if the source Markdown is correct, the publish step may:

- wrap headings in components,
- strip raw HTML ids,
- alter heading text during rendering,
- or generate routes that no longer match local assumptions.

At that point the anchor problem is no longer a source-only problem. It is a publishing-system problem.

## The safest troubleshooting order

Check in this order:

1. confirm the page itself loads,
2. inspect the rendered heading ID in the final output,
3. compare the fragment you wrote against the renderer's actual slug,
4. test whether a heading edit introduced the break,
5. then inspect whether the publish layer rewrote the page structure.

That catches most broken-anchor cases faster than repeatedly rewriting the link.

## The safest long-term habit

If anchor stability matters:

- keep headings simple,
- avoid unnecessary punctuation,
- avoid duplicate heading text when possible,
- and treat heading edits as link changes that need verification.

Markdown makes anchor links easy, but stable anchors still require editorial discipline.
