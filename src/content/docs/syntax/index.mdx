## The standard boundary

Two rules matter before anything else:

1. **CommonMark** defines the core syntax behavior that many modern renderers aim to follow.
2. **GitHub Flavored Markdown (GFM)** adds practical features such as tables and task lists that many people now expect, even though they are not all part of the core CommonMark spec.

That means a Markdown pattern may be:

- valid CommonMark,
- valid only in GitHub-style renderers,
- or tolerated by one tool but unreliable elsewhere.

If your file needs to travel, choose the form that survives the broadest path.

## The safest operating model

If you want Markdown that survives across repos, docs sites, static site generators, copy-paste workflows, and image export, favor these habits:

- use clear ATX headings unless you have a specific reason not to,
- use inline links for readability and reference links when reuse or repetition justifies them,
- use fenced code blocks instead of indented blocks in mixed-content documents,
- keep nested lists shallow and test task lists outside GitHub,
- treat tables as convenience structures, not universal layout containers,
- treat image sizing as a rendering decision unless portability is irrelevant,
- and assume line-break behavior is renderer-sensitive until tested.
