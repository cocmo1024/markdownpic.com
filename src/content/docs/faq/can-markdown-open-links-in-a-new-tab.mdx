# Can Markdown open links in a new tab?

## What matters first

No. **Standard Markdown does not have a native syntax for opening a link in a new tab.**

If you need that behavior, the solution usually comes from:

- raw HTML with `target="_blank"`,
- a renderer or docs-site component,
- or site-level JavaScript or template logic.

This is a behavior issue, not a Markdown syntax feature.

## Why this question keeps appearing

Links are one of the most common things people write in Markdown, so they expect basic browser behavior controls to be part of the same syntax.

But Markdown links are intentionally simple:

```md
[Visit the docs](https://example.com)
```

That syntax defines the destination. It does not define browser-window behavior.

## When HTML is the fallback

If the renderer allows raw HTML, the common fallback is:

```html
<a href="https://example.com" target="_blank" rel="noopener noreferrer">Visit the docs</a>
```

That can work, but it changes the tradeoff:

- you gain behavior control,
- you lose some portability,
- and you rely more heavily on the renderer not stripping or sanitizing the attributes.

## Why portability matters here

If the Markdown file needs to travel across:

- GitHub,
- docs sites,
- CMS fields,
- email,
- or export workflows,

then HTML attributes may not survive consistently.

That is why "open in new tab" is often better handled by the surrounding site or app instead of by the Markdown file itself.

## The better long-term rule

Use plain Markdown links by default.

Switch to HTML only when:

- the target renderer is known,
- the new-tab behavior is genuinely necessary,
- and the file does not need maximum portability.

If the same content must move through several systems, keep the source clean and let the publishing layer own the behavior.

## When not to force it

Do not force new-tab behavior into Markdown just because it is possible somewhere.

In many content systems, the healthier move is:

- keep the Markdown canonical,
- let the site theme or docs system decide external-link behavior,
- and avoid embedding browser-specific behavior into reusable source content.
