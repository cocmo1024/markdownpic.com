## What matters first

Markdown footnotes are useful, but they are not the safest portable syntax feature. Treat them as a renderer-dependent extension unless you control the output environment and know footnotes are supported.

## Standard example

```md
Here is a sentence with a footnote.[^1]

[^1]: This is the footnote text.
```

## Why footnotes feel deceptive

Footnotes look clean in well-supported renderers, so people assume they are universally safe. They are not.

The real issue is that footnotes sit in the zone where:

- some tools support them well,
- some tools ignore them,
- and some tools partially support them but produce awkward output in export.

That makes them different from headings, links, or plain lists.

## When footnotes are a good choice

Footnotes are reasonable when:

- the publishing environment is known,
- the audience benefits from reduced in-line clutter,
- the content is still primarily HTML or docs-page output,
- and you have tested the actual renderer.

They are especially defensible in long-form essays, technical notes, or editorial content where side comments would otherwise interrupt the flow.

## When footnotes are a poor choice

Footnotes are a poor choice when:

- the file must remain highly portable,
- the content will move through multiple unknown renderers,
- screenshots or image exports are important,
- or the note must remain easy to read in raw plain text.

## The fallback pattern

If footnotes are not safe enough for the workflow, use:

- a short parenthetical note,
- an end-of-section note block,
- or a â€śNotesâ€?subsection with normal links and references.

That is less elegant, but usually more stable.

## Recommended operating rule

Use footnotes only when the rendering path is known and the gain in readability is real. Otherwise, prefer normal prose and explicit note sections.
