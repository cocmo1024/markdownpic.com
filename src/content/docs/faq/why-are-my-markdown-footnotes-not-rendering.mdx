# Why are my Markdown footnotes not rendering?

## What matters first

Markdown footnotes often fail for one of four reasons:

1. your renderer does not support footnotes,
2. the reference and definition do not match,
3. the footnote syntax is malformed,
4. or the platform supports footnotes differently than your local preview.

The biggest hidden issue is that footnotes are not universal portable Markdown.

## Footnotes are not a guaranteed core Markdown feature

Many authors assume footnotes are part of standard Markdown everywhere.

They are not.

Some renderers support them well. Others ignore them, show them as plain text, or only support them in specific publishing surfaces.

That means the same source can look correct in one environment and broken in another.

## The most common syntax mistake

The most common mistake is mismatched footnote syntax:

```md
Here is a statement with a footnote.[^1]

[^1]: This is the note.
```

If the label in the reference and the label in the definition do not match exactly, the footnote will not resolve cleanly.

## Formatting mistakes also break footnotes

Footnotes often fail because of:

- missing colons,
- broken indentation,
- extra spaces in the wrong place,
- definitions placed where the renderer does not expect them,
- or copied Markdown that mixed renderer-specific styles.

Even small formatting drift can break the note.

## Renderer support is often the real issue

If the syntax looks correct but nothing renders, the problem is often the renderer.

A platform may:

- support footnotes in one Markdown surface but not another,
- support them in published docs but not in local preview,
- or require a specific parser or plugin.

That is why "it works on GitHub" and "it works in my editor" are not the same guarantee.

## The safest troubleshooting order

Check in this order:

1. confirm the reference and definition labels match exactly,
2. confirm the footnote definition syntax is correct,
3. simplify the example to one basic footnote,
4. check whether the target renderer actually supports footnotes,
5. then compare local preview against the real publishing environment.

That sequence catches most failures quickly.

## The healthiest long-term rule

Use footnotes only when the publishing environment is known to support them.

If the file must travel across many tools, treat footnotes as a renderer-dependent enhancement rather than guaranteed portable Markdown.
