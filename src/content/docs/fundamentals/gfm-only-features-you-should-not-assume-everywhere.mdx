## What matters first

Many of the Markdown features people now treat as "normal" are actually **GitHub Flavored Markdown conventions or renderer behavior**, not core portable Markdown.

The safest rule is:

- use GFM features when the render path is known,
- but do not let those features become accidental portability assumptions in canonical source.

## Why this matters

GitHub made several extensions feel mainstream because they are useful and familiar. The portability problem appears later when the same file moves into:

- docs generators,
- notes apps,
- email exports,
- static site pipelines,
- or image-generation workflows.

The file is still Markdown, but the feature contract has changed.

## High-frequency features people over-assume

These are among the most commonly over-assumed features:

- tables,
- task lists,
- strikethrough,
- autolink literals,
- alerts or callout-like constructs,
- footnotes in some ecosystems,
- and various HTML-tolerant patterns that "work on GitHub."

Some of these travel well. Others do not.

## The practical risk ranking

### Usually low-risk

- tables in many modern renderers,
- strikethrough in many common environments.

These are still not pure CommonMark guarantees, but the modern ecosystem often tolerates them reasonably well.

### Medium-risk

- task lists,
- autolink literals,
- footnotes,
- some image and heading-link behaviors.

These often work in several environments, but not enough to treat them as universal.

### High-risk

- GitHub alerts or callout syntax,
- custom containers,
- HTML-heavy layout tricks,
- renderer-specific components disguised as Markdown.

These are where portability usually fails first.

## A healthier authoring policy

For long-lived Markdown source:

1. treat CommonMark-safe content as the baseline,
2. allow selected GFM features intentionally,
3. document which extensions are approved,
4. and test the real render targets rather than assuming "Markdown is Markdown."

That policy matters more than memorizing a list.

## What usually belongs on the approved list

In many serious teams, the approved extension list ends up including:

- tables,
- strikethrough,
- maybe task lists,
- and a limited set of inline HTML escapes.

It usually excludes:

- renderer-specific alert syntax,
- layout hacks,
- and anything that makes the file hard to migrate or transform.
