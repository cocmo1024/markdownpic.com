# Can Markdown link to a heading in another file?

## What matters first

Yes, often.

A common pattern looks like this:

```md
[Go to installation](./guide.md#installation)
```

But that only works reliably when:

- the relative path resolves correctly,
- the target file keeps that route after publishing,
- and the renderer generates the heading anchor you expected.

So the syntax is simple. The portability is not.

## What has to work at the same time

A cross-file heading link combines **two** fragile things:

1. a file path,
2. and a fragment target.

The page may fail because the file path is wrong, the heading slug changed, or both.

## Why these links break so often

Cross-file heading links break when:

- the target heading text changes,
- the renderer changes slug rules,
- the file becomes a route like `/guide/` instead of `guide.md`,
- the docs framework strips or rewrites IDs,
- or the content moves into a system that no longer respects the original relative path.

That is why these links are powerful but not universally safe.

## The safest way to use them

Use cross-file heading links when:

- the content lives in a controlled docs system,
- headings are reasonably stable,
- and the team treats heading changes like link changes that need verification.

If the content is copied frequently across platforms, the links become much more brittle.

## The safest troubleshooting order

Check in this order:

1. does the target file load,
2. does the target heading still exist,
3. what slug did the final renderer generate,
4. did the publish layer rewrite the path,
5. did duplicate headings or punctuation change the fragment?

That sequence catches most failures quickly.

## When not to rely on them

Do not lean heavily on cross-file heading links when:

- the content is reused in multiple publishing systems,
- the docs framework auto-generates routes aggressively,
- or heading edits happen often and link maintenance is loose.

In those cases, page-level links or generated navigation may age better.
